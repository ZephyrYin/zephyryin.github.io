Web Scalability for Startup Engineers

Posted on 2024-05-05 Edited on 2024-06-06 In ReadingNotes

Core Concepts

What is Scalability?

“Scalability is an ability to adjust the capacity of the system to cost-efficiently fulfill the demands. Scalability usually means an ability to handle more users, clients, data, transactions, or requests without affecting the user experience. It is important to remember that scalability should allow us to scale down as much as scale up and that scaling should be relatively cheap and quick to do.”

  • Handling more data.
  • Handling higher concurrency levels.
  • Handling higher interaction rates.

Scalability vs Performance

  • Performance measures how long it takes to process a request or to perform a certain task.
  • Scalability measures how much we can grow (or shink).

Scaling Vertically: upgrade the hardware and network throughput.

  • Adding more I/O capacity by adding more hard drives
  • Improving I/O access times by switching to SSDs.
  • Reducing I/O operations by increasing RAM.
  • Improving network throughput by upgrading network interfaces or installing additional ones.
  • Switching to servers with more processors or more virtual cores.
    The limitations with scaling vertially:
  • Costy
  • There will always be a hard limit.
  • Operating system design or the application itself may prevent you from scaling beyond a certain point, (e.g lock contention).

Isolation of Servies: move different parts of the system to seperate physical servers.

“The core concept is that you should try to split your monilithic web application into a set of distinct functional parts and host them independently. The process of dividing a system based on functionality to scale it independently is called functional partitioning.”

Shift your workload to CNS

“A content delivery network is a hosted service that takes care of global distribution of static files like images and videos. Clients will download these files from CDN provider instead of your servers. If CDN server doesn’t have the requested content yet, it asks your server for it and caches it from then on.”

Scaling horizontally: Adding more servers whenever necessary.

Round-robin DNS: Map a domain name to multiple IP addresses, so that the client traffic is directed to different web servers.
Geo DNS: allow domain names to be resolved to IP addresses based on the location of the customer.

Principles of Good Software Design

Simplicity

Make things as simple as possible, but no spipler.

“Simplicity is not about using shortcuts and creating the quickest solution to the problem at hand. It is about what would be the easiest way for another software engineer to use your solution in the future.”
“Hidng complexity and building abstractions is one of the best ways to promote simplicity.”
“Good design allows you to add more details and features later on, but does not require you to build a massive solution up front. Beginning with a reasonable level of abstraction and iterating over it gives better results than trying to predit the future and build everything that might be needed later on.”

Loose Coupling

Loose coupling refers to a situation where different components know as little as nessary about each other

  • “Low coupling would allow you to introduce changes without the risk of breaking other parts of the system.”
  • “Low coupling promotes keeping complexity localized.”
  • “Decoupling on a higher level can mean having multiple applications, with each one focused on a narrow functionality. Then they can be scaled separately depending on their needs.”
    KeepDeclare as many methods as private/protected as possible.
    “Hiding details from outside world is a great way to reduce coupling and promote simplicity. Private methods are easier to be refactored, as the complexity is localized and you don’t need to search far to find out what could potentially break.”

Auto-Scaling

“Scalability is not just about scaling out, it is also about the ability to scale down, mainly to save cost.”

API-first

Priotize APIs as first class citizens, build your product around APIs. create a layer of web services that encapsulates most of the business logic and hides the complexcity behind a single API contract. Dedupe implementation for different clients(web, mobile…).

Data layer

ACID

The term ACID transaction refers to a set of transaction properties supported by most relational database engies.
-Atomic: An atomic transaction is executed in its entirely, it’s either completes or is rejected and reverted.
-Consistency: guarantees that every transaction transforms the data set from one consistent state to another and that once the transaction is complete, the data conforms to all of the constraints enforced by the data schema.
-Isolation guarantees that transactions can run in parallel without affecting each other.
-Durability guarantees that dat ais persisted before returning to the client, so that once a transaction is completed it can never be lost, even due to server failure.

CAP

-Consistency ensures all the nodes see the same data at the same time.
-Availability guarantees that any available node can serve client requests even when other nodes fail.
-Partition tolerance ensures that the system can operate even in the face of network failures where communication between nodes is impossible.

Eventually consistency

Amazon’s Dynamo data store is a classical eventual consistency system.

  • “Eventual consistency is a property of a system where different nodes may have different versions of the data, but where state changes eventually propagate to all of the servers. If you ask two servers for the same data at the same time in a global consistent system, it’s guaranteed to get the same result, while in a eventually consistent system, it’s not.”
